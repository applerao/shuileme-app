## 1. 数据库选型

### 1.1 数据库类型

| 数据库 | 版本 | 用途 | 说明 |
|--------|------|------|------|
| PostgreSQL | 15+ | 结构化数据 | 用户、测试、匹配、会员等需要事务的数据 |
| MongoDB | 7.0+ | 文档数据 | 日记、消息等灵活 schema 的数据 |
| Redis | 7.2+ | 缓存/会话 | 热点数据缓存、会话管理、队列 |

### 1.2 选型理由

**PostgreSQL:**
- ACID 事务保证
- 复杂查询支持
- 外键约束
- JSON 字段支持（混合场景）

**MongoDB:**
- 灵活 schema，适合日记内容变化
- 水平扩展能力强
- 文档模型，适合嵌套数据
- 全文搜索支持

**Redis:**
- 内存数据库，高性能
- 丰富数据结构
- 发布订阅支持
- 持久化可选

## 3. 表结构定义

### 3.1 PostgreSQL 表结构

#### 3.1.1 users (用户表)

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    phone VARCHAR(20) UNIQUE,
    avatar_url VARCHAR(500),
    gender SMALLINT DEFAULT 0,  -- 0: 未知，1: 男，2: 女，3: 其他
    birthday DATE,
    location VARCHAR(100),
    bio TEXT,
    personality_type VARCHAR(50),  -- MBTI 类型，如 INTJ
    personality_scores JSONB,  -- 各维度得分
    status SMALLINT DEFAULT 1,  -- 0: 禁用，1: 正常，2: 审核中
    last_login_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_username ON users(username);
CREATE INDEX idx_users_phone ON users(phone);
CREATE INDEX idx_users_personality_type ON users(personality_type);
CREATE INDEX idx_users_status ON users(status);
CREATE INDEX idx_users_location ON users(location);
```

#### 3.1.2 test_questions (测试题目表)

```sql
CREATE TABLE test_questions (
    id SERIAL PRIMARY KEY,
    category VARCHAR(50) NOT NULL,  -- E/I, S/N, T/F, J/P
    question_text TEXT NOT NULL,
    options JSONB NOT NULL,  -- [{"text": "选项 A", "score": 1}, ...]
    score_weights JSONB NOT NULL,  -- {"E": 1, "I": -1}
    sort_order INT DEFAULT 0,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_test_questions_category ON test_questions(category);
CREATE INDEX idx_test_questions_is_active ON test_questions(is_active);
CREATE INDEX idx_test_questions_sort_order ON test_questions(sort_order);
```

#### 3.1.3 personality_tests (性格测试结果表)

```sql
CREATE TABLE personality_tests (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    test_version VARCHAR(20) DEFAULT '1.0',
    answers JSONB NOT NULL,  -- [{question_id, answer, score}]
    result_type VARCHAR(10) NOT NULL,  -- 如 INTJ
    result_scores JSONB NOT NULL,  -- {"E": 60, "I": 40, ...}
    result_description TEXT,
    report_url VARCHAR(500),
    completed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_personality_tests_user_id ON personality_tests(user_id);
CREATE INDEX idx_personality_tests_result_type ON personality_tests(result_type);
CREATE INDEX idx_personality_tests_completed_at ON personality_tests(completed_at);

-- 唯一约束：每个用户只保留最新测试结果
CREATE UNIQUE INDEX idx_personality_tests_user_latest 
    ON personality_tests(user_id, completed_at DESC) 
    WHERE completed_at IS NOT NULL;
```

#### 3.1.4 matches (匹配表)

```sql
CREATE TABLE matches (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    matched_user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    match_score DECIMAL(5,2) NOT NULL,  -- 匹配度 0-100
    match_reasons JSONB,  -- 匹配原因 [{"type": "personality", "description": "..."}]
    status SMALLINT DEFAULT 0,  -- 0: 待响应，1: 喜欢，2: 不喜欢，3: 已匹配
    initiator_id UUID NOT NULL REFERENCES users(id),  -- 发起方
    responded_at TIMESTAMP,
    matched_at TIMESTAMP,  -- 双方都喜欢时记录
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    CONSTRAINT chk_match_score CHECK (match_score >= 0 AND match_score <= 100),
    CONSTRAINT chk_match_users_different CHECK (user_id != matched_user_id)
);

-- 索引
CREATE INDEX idx_matches_user_id ON matches(user_id);
CREATE INDEX idx_matches_matched_user_id ON matches(matched_user_id);
CREATE INDEX idx_matches_status ON matches(status);
CREATE INDEX idx_matches_match_score ON matches(match_score DESC);
CREATE INDEX idx_matches_created_at ON matches(created_at DESC);

-- 唯一约束：防止重复匹配
CREATE UNIQUE INDEX idx_matches_unique_pair 
    ON matches(LEAST(user_id, matched_user_id), GREATEST(user_id, matched_user_id));
```

#### 3.1.5 subscriptions (会员表)

```sql
CREATE TABLE subscriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    plan_type VARCHAR(20) NOT NULL,  -- monthly, quarterly, yearly
    plan_name VARCHAR(50) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    currency VARCHAR(10) DEFAULT 'CNY',
    start_date DATE NOT NULL,
    end_date DATE NOT NULL,
    status SMALLINT DEFAULT 1,  -- 0: 取消，1: 有效，2: 过期，3: 退款中
    payment_method VARCHAR(20),  -- wechat, alipay, apple_pay
    payment_transaction_id VARCHAR(100),
    auto_renew BOOLEAN DEFAULT false,
    cancelled_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_subscriptions_user_id ON subscriptions(user_id);
CREATE INDEX idx_subscriptions_status ON subscriptions(status);
CREATE INDEX idx_subscriptions_end_date ON subscriptions(end_date);
CREATE INDEX idx_subscriptions_plan_type ON subscriptions(plan_type);
```

### 3.2 MongoDB Collection 结构

#### 3.2.1 diaries (日记集合)

```javascript
{
  _id: ObjectId,
  user_id: UUID,  // 关联 PostgreSQL users.id
  title: String,
  content: String,
  mood: String,  // emoji，如 😊 😔 😠
  tags: [String],
  images: [
    {
      url: String,
      thumbnail: String,
      uploaded_at: Date
    }
  ],
  ai_suggestions: {
    writing_help: String,
    polish_version: String,
    emotion_analysis: {
      primary_emotion: String,
      score: Number,
      description: String
    }
  },
  emotion_score: Number,  // -100 到 100
  is_public: Boolean,
  view_count: Number,
  like_count: Number,
  comment_count: Number,
  location: {
    type: "Point",
    coordinates: [longitude, latitude]
  },
  weather: {
    condition: String,
    temperature: Number
  },
  created_at: Date,
  updated_at: Date
}

// 索引
db.diaries.createIndex({ user_id: 1, created_at: -1 });
db.diaries.createIndex({ is_public: 1, created_at: -1 });
db.diaries.createIndex({ tags: 1 });
db.diaries.createIndex({ mood: 1 });
db.diaries.createIndex({ location: "2dsphere" });  // 地理位置查询
```

#### 3.2.2 messages (消息集合)

```javascript
{
  _id: ObjectId,
  conversation_id: String,  // 会话 ID，格式："user1_user2"（按字母排序）
  sender_id: UUID,
  receiver_id: UUID,
  content: String,
  type: String,  // text, image, voice, system
  media: {
    url: String,
    thumbnail: String,
    duration: Number,  // 语音/视频时长（秒）
    size: Number  // 文件大小（字节）
  },
  status: String,  // sent, delivered, read
  read_at: Date,
  reply_to: ObjectId,  // 回复的消息 ID
  created_at: Date,
  updated_at: Date

// 索引
db.messages.createIndex({ conversation_id: 1, created_at: -1 });
db.messages.createIndex({ sender_id: 1, created_at: -1 });
db.messages.createIndex({ receiver_id: 1, created_at: -1 });
db.messages.createIndex({ created_at: -1 }, { expireAfterSeconds: 2592000 });  // 30 天 TTL（可选）
```

### 3.3 Redis 数据结构

```
# 会话管理
session:{user_id} = {user_info_json}  # TTL: 7 天

# 用户信息缓存
cache:user:{user_id} = {user_info_json}  # TTL: 30 分钟
cache:user:username:{username} = {user_id}  # TTL: 30 分钟

# 匹配推荐缓存
cache:matches:{user_id} = [{matched_user_id, score}]  # TTL: 1 小时

# 在线状态
online:{user_id} = timestamp  # TTL: 5 分钟（心跳续期）

# 限流
ratelimit:api:{user_id}:{endpoint} = count  # TTL: 1 分钟

# 消息队列
queue:notifications = [notification_json, ...]  # List
queue:ai_tasks = [task_json, ...]  # List

# 发布订阅
channel:user:{user_id}  # 用户消息推送
channel:system  # 系统广播
```

## 4. 索引设计

### 4.1 PostgreSQL 索引策略

| 表名 | 索引名 | 字段 | 类型 | 用途 |
|------|--------|------|------|------|
| users | idx_users_email | email | B-Tree | 登录查询 |
| users | idx_users_username | username | B-Tree | 用户查找 |
| users | idx_users_personality_type | personality_type | B-Tree | 匹配筛选 |
| personality_tests | idx_personality_tests_user_id | user_id | B-Tree | 获取用户测试结果 |
| matches | idx_matches_user_id | user_id | B-Tree | 获取用户匹配 |
| matches | idx_matches_status | status | B-Tree | 待处理匹配查询 |
| subscriptions | idx_subscriptions_user_id | user_id | B-Tree | 会员状态查询 |
| user_actions | idx_user_actions_user_id | user_id | B-Tree | 行为日志查询 |

### 4.2 MongoDB 索引策略

| Collection | 索引字段 | 类型 | 用途 |
|-----------|---------|------|------|
| diaries | user_id + created_at | Compound | 用户日记列表 |
| diaries | is_public + created_at | Compound | 公开日记列表 |
| diaries | tags | Multikey | 标签搜索 |
| diaries | location | 2dsphere | 地理位置查询 |
| messages | conversation_id + created_at | Compound | 会话消息列表 |

### 4.3 索引优化建议

1. **避免过度索引**: 每个索引增加写入开销
2. **复合索引顺序**: 高频查询字段放前面
3. **定期分析**: 使用 `EXPLAIN ANALYZE` 分析查询性能
4. **监控慢查询**: 记录执行时间 > 100ms 的查询

## 6. 数据迁移策略

### 6.1 迁移工具

- PostgreSQL: `pg_dump` / `pg_restore`
- MongoDB: `mongodump` / `mongorestore`
- 应用层迁移：使用 Sequelize/Mongoose migration

### 6.2 迁移流程

```
1. 备份现有数据
   ↓
2. 在测试环境验证迁移脚本
   ↓
3. 停机维护公告
   ↓
4. 执行迁移脚本
   ↓
5. 数据验证（记录数、关键字段）
   ↓
6. 回滚预案准备
   ↓
7. 上线验证
   ↓
8. 恢复服务
```

### 6.3 回滚策略

- 迁移前完整备份
- 迁移脚本可逆（up/down）
- 分阶段迁移，每阶段可独立回滚

## 7. 性能优化

### 7.1 查询优化

```sql
-- 使用 EXPLAIN 分析查询计划
EXPLAIN ANALYZE 
SELECT * FROM users 
WHERE personality_type = 'INTJ' 
  AND status = 1 
  AND location = '北京';

-- 优化建议：
-- 1. 添加复合索引 (personality_type, status, location)
-- 2. 避免 SELECT *，只查询需要的字段
-- 3. 使用 LIMIT 限制返回数量
```

### 7.2 连接池配置

```javascript
// PostgreSQL 连接池
const pool = new Pool({
  max: 20,  // 最大连接数
  min: 5,   // 最小连接数
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

// MongoDB 连接池
mongoose.connect(uri, {
  maxPoolSize: 50,
  minPoolSize: 10,
  maxIdleTimeMS: 30000,
});
```

### 7.3 读写分离

```
主库 (Write): 用户注册、登录、匹配操作
从库 (Read):  日记列表、用户信息、匹配推荐

配置：
- 主库：1 台（高配置）
- 从库：2 台（只读复制）
- 复制延迟监控：< 1 秒
```

## 8. 安全设计

### 8.1 数据加密

| 数据类型 | 加密方式 | 说明 |
|---------|---------|------|
| 密码 | bcrypt | salt rounds = 10 |
| 敏感字段 | AES-256 | 手机号、支付信息 |
| 传输 | TLS 1.3 | HTTPS/WSS |
| 日记内容 | 可选端到端加密 | 用户密钥 |

### 8.2 访问控制

```sql
-- PostgreSQL 行级安全策略 (RLS)
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

CREATE POLICY user_isolation ON users
  USING (id = current_setting('app.current_user_id')::uuid);
```

### 8.3 数据脱敏

```javascript
// 日志中的敏感数据脱敏
function maskSensitiveData(data) {
  return {
    ...data,
    email: data.email?.replace(/(.{2}).+(@.+)/, '$1***$2'),
    phone: data.phone?.replace(/(.{3}).+(.{4})/, '$1****$2'),
  };
}
```

## 9. 备份与恢复

### 9.1 备份策略

| 数据库 | 备份频率 | 保留周期 | 备份方式 |
|--------|---------|---------|---------|
| PostgreSQL | 每日全量 + 每小时增量 | 30 天 | pg_dump + WAL 归档 |
| MongoDB | 每日全量 | 14 天 | mongodump |
| Redis | 每小时 RDB | 7 天 | RDB 快照 |

### 9.2 备份脚本示例

```bash
#!/bin/bash
# PostgreSQL 备份脚本

BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="xinyu"

# 全量备份
pg_dump -h localhost -U admin -d $DB_NAME -F c -f ${BACKUP_DIR}/xinyu_${DATE}.dump

# 清理 30 天前的备份
find ${BACKUP_DIR} -name "*.dump" -mtime +30 -delete
```

### 9.3 恢复测试

- 每月进行一次恢复演练
- 验证备份完整性
- 记录恢复时间目标 (RTO)

## 10. 监控指标

### 10.1 数据库监控

| 指标 | 告警阈值 | 说明 |
|------|---------|------|
| 连接数使用率 | > 80% | 连接池耗尽风险 |
| 慢查询比例 | > 5% | 查询性能下降 |
| 复制延迟 | > 5 秒 | 主从同步问题 |
| 磁盘使用率 | > 80% | 存储空间不足 |
| QPS | 突增 50% | 异常流量 |

| 指标 | 告警阈值 | 说明 |
|------|---------|------|
| API 响应时间 P95 | > 500ms | 接口性能下降 |
