## 3. 数据库设计

### 3.1 数据库选型原则

- **PostgreSQL**: 用于结构化数据，需要事务保证的场景（用户、支付、匹配）
- **MongoDB**: 用于非结构化/半结构化数据，频繁读写的场景（日记、消息）
- **Redis**: 用于缓存、会话管理、实时数据

### 3.2 核心数据表概览

| 表名 | 数据库 | 说明 |
|------|--------|------|
| users | PostgreSQL | 用户基本信息 |
| personality_tests | PostgreSQL | 性格测试题目与结果 |
| diaries | MongoDB | 用户日记 |
| matches | PostgreSQL | 用户匹配关系 |
| messages | MongoDB | 聊天消息 |
| subscriptions | PostgreSQL | 会员订阅信息 |

## 4. API 设计规范

### 4.1 RESTful 设计原则

- 使用 HTTP 动词：GET (查询), POST (创建), PUT (更新), DELETE (删除)
- 资源命名使用复数名词：`/api/users`, `/api/diaries`
- 版本号包含在路径中：`/api/v1/`
- 统一响应格式

### 4.2 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { },
  "timestamp": 1709999999999
}
```

### 4.3 错误码规范

| 错误码范围 | 说明 |
|-----------|------|
| 200 | 成功 |
| 400-499 | 客户端错误 |
| 500-599 | 服务端错误 |

### 4.4 认证机制

- 使用 JWT Token 进行身份认证
- Token 有效期：7 天
- Refresh Token 有效期：30 天
- Token 存储于 HTTP Only Cookie 或本地存储

### 4.5 限流策略

- 普通接口：100 次/分钟/IP
- 敏感接口（登录、注册）：10 次/分钟/IP
- AI 接口：30 次/分钟/用户

*详细 API 设计见 `API 文档.md`*

## 5. 安全设计

### 5.1 认证与授权

```
┌─────────────────────────────────────────────────────────────┐
│                    认证授权流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户登录 → 验证凭证 → 生成 JWT → 返回 Token                 │
│      ↓                                                      │
│  后续请求 → 携带 Token → 验证签名 → 解析用户信息 → 授权访问   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 数据安全

| 安全措施 | 实现方式 |
|---------|---------|
| 密码存储 | bcrypt 加密，salt rounds=10 |
| 传输加密 | HTTPS (TLS 1.3) |
| 敏感数据 | 数据库字段级加密 |
| 日记隐私 | 仅用户本人可见，端到端加密 (可选) |

### 5.3 接口安全

- CSRF 防护：SameSite Cookie 策略
- XSS 防护：输入输出过滤，CSP 策略
- SQL 注入防护：参数化查询，ORM 使用
- 请求签名：敏感接口添加签名验证

### 5.4 隐私保护

- 用户数据最小化收集
- 日记内容加密存储
- 匿名化数据分析
- GDPR/个人信息保护法合规

### 5.5 安全审计

- 登录日志记录
- 敏感操作审计
- 异常行为监控
- 定期安全扫描

## 6. 扩展性设计

### 6.1 水平扩展架构

```
┌─────────────────────────────────────────────────────────────┐
│                    水平扩展架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户量增长 → 增加应用实例 → 负载均衡分发                    │
│      ↓                                                      │
│  数据量增长 → 数据库读写分离 → 分库分表                      │
│      ↓                                                      │
│  流量增长 → CDN 加速 → 静态资源缓存                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 微服务拆分策略

| 阶段 | 服务粒度 | 说明 |
|------|---------|------|
| 初期 (0-10 万用户) | 单体应用 | 快速迭代，降低复杂度 |
| 中期 (10-50 万用户) | 模块化单体 | 按业务模块拆分 |
| 后期 (50 万 + 用户) | 微服务架构 | 独立部署，弹性伸缩 |

### 6.3 缓存策略

| 缓存层级 | 缓存内容 | 过期策略 |
|---------|---------|---------|
| L1 (Redis) | 用户信息、配置 | 5-30 分钟 |
| L2 (应用内存) | 热点数据 | 1-5 分钟 |
| L3 (CDN) | 静态资源 | 7-30 天 |

### 6.4 消息队列

- 使用 Redis Stream 或 RabbitMQ
- 异步任务：短信发送、邮件通知、AI 处理
- 削峰填谷：高并发场景下的请求缓冲

### 6.5 数据库扩展

- 读写分离：主库写，从库读
- 分库分表：用户 ID 哈希分片
- 数据归档：历史数据迁移到冷存储

## 10. 风险评估

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| AI API 调用成本过高 | 中 | 高 | 缓存结果、限流、降级策略 |
| 用户数据泄露 | 低 | 高 | 加密存储、访问控制、审计日志 |
| 服务器宕机 | 中 | 高 | 多实例部署、自动重启、监控告警 |
| 匹配算法效果不佳 | 中 | 中 | A/B 测试、用户反馈迭代 |
| 开发进度延期 | 中 | 中 | 敏捷开发、优先级排序、加班预案 |
